\begin{abstract}
Este relatório descreve a implementação de um sistema de serviço de registo de eventos em séries temporais e de agregação de informação, relativos a venda de produtos. 
  A informação é mantida num \emph{Servidor} e acedida remotamente através de um \emph{Cliente} conectado por sockets TCP.

  \keywords{TCP \and Serialização TLV \and Java Concurrency.}
\end{abstract}

\section{Introdução}
O sistema desenvolvido baseia-se numa arquitetura cliente-servidor, onde a comunicação é estabelecida através de sockets TCP. O servidor foi desenhado de forma a processar pedidos de multiplos clientes concurrentemente, pelo que foi necessário cuidados para garantir a sincronização de dados dentro das várias estruturas do servidor.
Esta solução proposta visa uma correção eficiente, valorizando estratégias que minimizam o número de threads ativas.

\section{Pedidos}
Os pedidos dos clientes ao servidor são realizados através do protocolo TCP, garantindo fiávelmente que o pedido e a sua resposta sejam recebidas.

Na interface do cliente todos os pedidos são \emph{case-insensitive} de forma a facilitar a escrita e apelar a todos os utilizadores.
Um cliente realiza um pedido através da interface dada, estes pedidos podem ser dos tipos:

\subsection{Autenticação}
Para poder realizar pedidos um utilizador tem, obrigatoriamente, de estar autenticado. Para este fim são disponibilizados os pedidos de \texttt{Register} e \texttt{Login}.
Ambos os pedidos necessitam de dois campos para serem válidos.

\begin{itemize}
  \item \emph{Username}: O nome pelo qual o utilizador será autenticado.
  \item \emph{Password}: A palavra-passe do utilizador.
\end{itemize}

Caso já autenticado um utilizador não poderá se autenticar novamente nem registar nenhum utilizador.


\subsection{Registo de Eventos}
O registo de um evento relativo ao dia corrente é feito através do pedido \texttt{ADD\_SALE}.
Este pedido será executado corretamente apenas quando o produto ao qual se pretende adicionar um evento existe, pelo que caso contrário nenhuma alteração será feita. Para este efeito são necessários os argumentos.

\begin{itemize}
  \item \emph{ProductName}: O nome do produto ao qual se pretende adicionar um evento.
  \item \emph{Quantity}: A quantidade de produtos vendida. 
  \item \emph{SellPrice}: O preço pelo qual os produtos foram vendidos.
\end{itemize}

\subsection{Queries}
Um cliente pode realizar \emph{Queries} sobre os dados do servidor.
Todas as \emph{queries} disponibilizadas necessitam dos mesmos dados para serem executadas corretamente:
\begin{itemize}
  \item \emph{ProductName}: O nome do produto sobre o qual se realiza a query.
  \item \emph{NumDays}: O número de dias posteriores ao atual sobre o qual se realiza a query.
\end{itemize}

\noindent Um cliente pode fazer pedidos relativos à quantidade de que um produto foi vendido, ao preço máximo pelo qual foi vendido, o preço médio e ainda a receita total das vendas desse produto.
Para isto são usados respetivamente os pedidos:

\noindent \texttt{Query\_QTD}
\\\texttt{Query\_MAX}
\\\texttt{Query\_AVG}
\\\texttt{Query\_TOTAL}

\subsection{Filtros}
Os pedidos de filtros são pedidos que dado um conjunto de itens e um dia anterior ao corrente, devolve uma lista de todos os eventos desses itens de forma organizada e compacta.
\begin{itemize}
  \item \emph{ProductNames}: Nomes dos produtos
  \item \emph{Day}: Dia sobre o qual se quer os eventos

\section{Pacotes}
Um pacote, definido em \texttt{Packet.java} é uma estrutura geral para enviar informação entre cliente e servidor. 
\\ Este é definido pelo seu \emph{header} e \emph{data}. 

\subsection{Header}
Definido em \texttt{comms::common::PacketHeader.java}, um header de pacote contém informação comum a todos os pacotes enviados.

\begin{itemize}
  \item \emph{PacketSize}: O tamanho total do pacote, em bytes. \texttt{Integer}
  \item \emph{PacketID}: O identificador único do pacote. \texttt{Integer}
  \item \emph{ClientID}: O identificador único do cliente, -1 caso servidor. \texttt{Integer}
  \item \emph{Type}: O tipo do pacote. \texttt{Byte}
\end{itemize}

\noindent
\\ Como todos os header contêm os mesmo campos, todos de tamanho fixo, este tem sempre o mesmo tamanho de 13 bytes. Isto facilita a leitura do pacote, visto que todos os bytes após o décimo-terceiro serão bytes de \emph{payload}.

\section{Serialização}
De forma a enviar dados eficientemente estes são serializados pelo \emph{encoder} desenvolvido \texttt{comms::common::Encodable.java}. Esta classe é abstrata, pelo que deve ser implementada por estruturas que necessitam de serializar dados, neste caso, \texttt{Packet.java e PacketHeader.java}.
\\
Numa abordagem inicial foi desenvolvido um serializador do tipo \texttt{TLV}, \emph{Type-Length-Value}, no entanto como a leitura dos dados não necessita de tipos de dados variáveis optou-se por uma abordagem mais simples largando o \emph{Type} e utilizando \emph{Length} apenas quando necessário como por exemplo para tamanhos de \emph{strings}.

\subsection{Pedidos}
Os pedidos que um cliente faz ao servidor são realizados através da interface. Após ser registado um pedido este é validado e caso válido começa o processo de realização.

Para gerir os pedidos de um cliente foi desenvolvida uma estrutura de threads semelhante a uma \emph{ThreadPool} tradicional. Ao ser registado um pedido procura-se por threads já criadas que estejam paradas, caso exista alguma é utilizada essa thread para enviar o pedido e esperar a sua resposta. Caso todas as threads estejam ocupadas, ou ainda não existe nenhuma, é criada uma nova para realizar esta função.

Após ser enviado o pacote ao servidor, a thread responsável fica bloqueada até receber uma resposta ao ser pedido, quando então comunica a resposta ao cliente.
\\
Como podem existir múltiplas threads à espera de concluir o seu pedido, ao este ser enviado é registado o seu identificador único. Apartir deste, ao ser recebido uma resposta a um pedido acorda-se a thread que está bloqueada à espera desse identificador, efetivamente funcionando como um \emph{demultiplexador}.

\subsection{Conexão}
A conexão cliente-servidor é feita através de sockets TCP. Para gerir esta conexão é utilizada uma thread \texttt{ClientConnectionThread.java}. Esta thread usa uma classe geral \texttt{Connection.java}, que contém métodos de enviar e receber pacotes, para gerir a conexão. 
\\ Quando recebido um pacote é verificado se este é válido e se era esperado. Caso isto, é lido o identificador do pacote e enviado um \emph{signal} à thread que o espera.

A interface proporcionada ao cliente facilita a realização de operações, mostrando ao utilizador as operações disponiveis e as respetivas inputs, garantindo a sanitação das mesmas.


\section{Servidor}
O servidor armazena em disco todos os dados necessários para o seu funcionamento. 
Após ser iniciado, este tem sempre uma thread à espera que clientes façam a conexão através da socket TCP, após um cliente se conectar é criada uma thread que irá controlar a socket de conexão com esse cliente, recebendo pedidos e enviando a sua resposta.

\subsection{Pedidos}
Após um cliente se conectar ao servidor este necessita, obrigatoriamente, de realizar um pedido de \emph{login}, apenas após este será dado um identificador único ao cliente, que o deixará realizar pedidos.

Quando uma thread \texttt{ClientHandlerThread}, definida em \texttt{server::ClientHandlerThread.java}, recebe um pedido do cliente esta cria outra thread \texttt{RequestHandlerThread} que irá executar o pedido dependendo do seu tipo. 
Caso o pedido seja executado com êxito, a \texttt{RequestHandlerThread} irá enviar o pacote de resposta através da conexão gerida pela \texttt{ClientHandlerThread}.

\subsection{Eventos}
Os produtos mantidos no servidor são utilizados como estrutura base dos outros componentes, todos os produtos são caracterizados por

\begin{itemize}
  \item \emph{ID}: Identificador Único. \texttt{Integer}
  \item \emph{Name}: Nome do produto. \texttt{String}
\end{itemize}

Todos os produtos existentes no sistema são guardados pelo servidor de forma persistente. A utilização de um identificador facilita a realização de cálculos e a armazenação de informação pertinente aos produtos.

\noindent Os eventos são sempre relativos a um produto, representando uma venda deste caracterizada por

\begin{itemize}
  \item \emph{SellPrice}: Preço a qual foi vendido. \texttt{Float}
  \item \emph{Quantity}: Quantidade vendida. \texttt{Integer}
\end{itemize}


\subsection{Dias}
Os dias são representados no servidor pela classe \texttt{WorkDay}, que contém estruturas e métodos necessários para realizar as operações disponíveis aos clientes.
Um \emph{WorkDay} refere-se sempre a uma data, pelo que não podem existir dois \emph{WorkDays} com a mesma data, efetivamente fazendo o papel de identificador único.

\noindent Esta estrutura guarda informação relativa à venda de produtos, guardando todos os eventos relativos a aos produtos que foram realizados no dia.
Para facilitar o manuseamento destes dados foi desenvolvida a classe \texttt{ProductEntry}

\begin{itemize}
  \item \emph{ProductID}: O produto a qual se refere. \texttt{Integer}
  \item \emph{Events}: Os eventos relativos ao produto. \texttt{List<Event>}
\end{itemize}

Esta estrutura contém também operações necessárias para realizar cálculos sobre os eventos. Visto que os eventos são efetivamente imutáveis após o dia mudar, estes cálculos são colocados em cache enquanto o dia em si permanecer na memória.

\noindent Como ambos \emph{WorkDay} e \emph{ProductEntry} esperam múltiplos clientes realizar operações sobre eles, estes contém mecanismos de controlo de concurrência utilizando \texttt{Locks}.
